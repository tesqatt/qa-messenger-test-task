# Тестовое задание на позицию QA Engineer (Middle+/Senior)

## Общее описание

В данном репозитории находятся:
- краткий анализ сценария
- чек-лист для ручного тестирования
- 4 тест-кейса
- описание API-части
- один E2E-тест на Playwright для основного сценария отправки сообщения

Сценарий задания относится к веб-клиенту мессенджера, где пользователь должен иметь возможность:
- открыть существующую комнату
- отправить текстовое сообщение
- увидеть это сообщение в ленте

API-эндпоинт в рамках задания:

`PUT /_matrix/client/v3/rooms/{roomId}/send/{eventType}/{txnId}`

Для обычного текстового сообщения:
- `eventType = m.room.message`

## Допущения

В описании задания не указано конкретное публичное окружение или URL приложения, поэтому я рассматривал это тестовое как задачу на тест-дизайн, понимание API-контракта и демонстрацию подхода к автоматизации.

Для E2E-теста я исходил из следующих допущений:
- в приложении есть страница логина
- пользователь может открыть существующую комнату
- в комнате есть поле ввода сообщения и возможность отправки
- после успешной отправки сообщение отображается в ленте

Также я считаю, что успешный результат это не только успешный ответ от сервера, но и фактическое отображение сообщения в ленте для пользователя.

## Как запускать

Установка зависимостей:

```bash

```

Запуск тестов:

```bash
npx playwright test
```

## Переменные окружения

В тесте используются следующие переменные окружения:

```bash
BASE_URL=https://example.com
TEST_USER_EMAIL=test@example.com
TEST_USER_PASSWORD=secret
TEST_ROOM_NAME=QA Test Room
TEST_MESSAGE=Hello from Playwright
```

`https://example.com` указан как placeholder, так как в задании не было дано конкретного тестового окружения.

Если в приложении используется логин по username, а не по email, селекторы и названия переменных можно скорректировать.

## Комментарий по автоматизации

По условиям задания требовалось написать один E2E-тест, поэтому я сосредоточился на основном стабильном happy path сценарии:
- логин
- открытие комнаты
- отправка сообщения
- проверка, что сообщение отображается в ленте

Если бы было доступно реальное тестовое окружение проекта, дополнительно я бы предложил:
- добавить стабильные `data-testid` для критичных элементов
- выделить отдельного тестового пользователя
- выделить отдельную тестовую комнату
- предусмотреть подготовку или сброс тестовых данных

Это сделало бы E2E-тест более стабильным и удобным для поддержки.
