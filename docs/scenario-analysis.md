# Краткий анализ сценария

## Что я считаю объектом тестирования

В рамках этого задания я рассматриваю объект тестирования как сквозной сценарий отправки текстового сообщения в веб-клиенте мессенджера.

В него входят несколько уровней:

- **UI-уровень** — логин, открытие комнаты, поле ввода сообщения, действие отправки, отображение сообщения в ленте.
- **Клиентская логика** — формирование запроса, отправка запроса с авторизацией, обработка успешных и ошибочных ответов, обновление интерфейса.
- **API-уровень** — `PUT /_matrix/client/v3/rooms/{roomId}/send/{eventType}/{txnId}`.
- **Серверная часть** — проверка авторизации, проверка существования комнаты, создание события сообщения, обработка идемпотентности по `txnId`.

То есть я рассматриваю не только сам API, но и пользовательский результат целиком.  
С точки зрения пользователя сценарий можно считать успешным только в том случае, если сообщение не просто принято сервером, а действительно появилось в ленте комнаты.

Если сервер вернул успешный ответ, но пользователь не увидел сообщение в интерфейсе, я бы не считал такой сценарий полностью успешным.

## Основные риски в сценарии отправки сообщения

### 1. Сообщение принято сервером, но не отображается в UI
API может вернуть успех, но клиент может не обновить ленту или некорректно отрисовать новое сообщение.

### 2. Появление дубликатов
Это один из ключевых рисков, так как в задании отдельно упоминается `txnId`.  
Если идемпотентность реализована некорректно, при повторном запросе может создаться дубликат сообщения.

### 3. Проблемы с авторизацией
Токен может отсутствовать, быть невалидным или истекшим.  
В этом случае система должна корректно отклонять запрос, а клиент не должен показывать ложный успешный результат.

### 4. Отправка в несуществующую или недоступную комнату
Комната может не существовать, быть удалена или быть недоступна пользователю.

### 5. Рассинхронизация между клиентом и сервером
Клиент может отображать сообщение оптимистично до получения ответа сервера. Если запрос завершится ошибкой, состояние UI может стать неконсистентным.

### 6. Граничные случаи по содержимому сообщения
Даже для обычного текстового сообщения важны такие случаи, как:
- пустое сообщение;
- сообщение только из пробелов;
- длинный текст;
- многострочный текст;
- специальные символы и emoji.

### 7. Повторная отправка из-за сетевых проблем
При нестабильной сети клиент может повторить запрос. Именно в таких ситуациях ошибка в обработке `txnId` может привести к дублированию сообщения.

## Граничные случаи, на которые я бы обратил внимание

Для данного сценария я бы отдельно проверял:
- существующую и несуществующую комнату;
- валидный и невалидный / истекший токен;
- первый запрос и повторный запрос с тем же `txnId`;
- пустое сообщение;
- сообщение только из пробелов;
- длинное сообщение;
- многострочный текст;
- Unicode / emoji;
- замедленный ответ сервера;
- повторную отправку по инициативе пользователя.

## Допущения

Так как требования сформулированы кратко, я делаю следующие допущения:

1. Успешный результат — это одновременно:
   - сервер принял сообщение;
   - пользователь увидел сообщение в ленте комнаты.

2. Для обычного текстового сообщения используется:
   - `eventType = m.room.message`
   - body с текстом сообщения и типом сообщения.

3. Для запроса требуется валидная авторизация по токену.

4. `txnId` используется для идемпотентности:
   повторный запрос с тем же `txnId` не должен создавать новое сообщение.

5. Комната уже существует, если отдельно не оговорено создание новой комнаты.

6. Если отправка завершилась ошибкой, клиент не должен вести себя так, как будто сообщение было успешно доставлено.
