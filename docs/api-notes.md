# API-часть

## 1. Как должен выглядеть запрос на отправку текстового сообщения

Эндпоинт для отправки сообщения:

`PUT /_matrix/client/v3/rooms/{roomId}/send/{eventType}/{txnId}`

Для отправки обычного текстового сообщения я ожидаю следующие параметры:
- `roomId` - идентификатор комнаты
- `eventType = m.room.message`
- `txnId` - уникальный идентификатор транзакции, который клиент использует для идемпотентности

### Ожидаемое тело запроса (body)

Минимально я ожидаю, что в теле запроса будут:
- тип сообщения;
- текст сообщения.

Пример минимального body:

```json
{
  "msgtype": "m.text",
  "body": "Hello QA"
}
```

Где:
- `msgtype` показывает, что это обычное текстовое сообщение
- `body` содержит сам текст, который должен увидеть пользователь

### Ожидаемые заголовки

Минимально я ожидаю такие заголовки:
- `Authorization: Bearer <access_token>`
- `Content-Type: application/json`

### Авторизация

Для выполнения запроса клиент должен передавать валидный access token.

Если токен отсутствует, невалиден или истёк, сервер должен отклонить запрос, а клиент не должен считать сообщение успешно отправленным.

---

## 2. Какие ответы сервера я ожидаю

### Успешная отправка

При успешной отправке клиенту нужен минимум данных, чтобы понять, что сообщение принято сервером и связано с конкретным событием.

Я ожидаю:
- успешный HTTP-статус
- идентификатор созданного события в response body

Например, в упрощённом виде:

```json
{
  "event_id": "$some_event_id"
}
```

Этого достаточно, чтобы клиент мог связать успешную отправку с конкретным сообщением и корректно обновить интерфейс.

### Несуществующая комната

Если комнаты не существует, я ожидаю:
- клиентскую ошибку
- понятный ответ с описанием проблемы
- отсутствие созданного события

Точный статус-код может зависеть от реализации, но по смыслу ответ должен показывать, что отправка в указанную комнату невозможна.

### Отсутствующий, невалидный или истёкший токен

Если токен отсутствует, невалиден или истёк, я ожидаю:
- ошибку аутентификации или авторизации
- понятный ответ от сервера
- отсутствие созданного сообщения

С точки зрения клиента важно, чтобы такая ситуация не выглядела как успешная отправка. Пользователь должен либо увидеть ошибку, либо получить запрос на повторную авторизацию.

## 3. Как я бы проверял идемпотентность по txnId

Ключевое требование в этом сценарии - повторный запрос с тем же `txnId` не должен создавать новое сообщение.

### Подход к проверке

1. Подготовить валидный запрос:
   - тот же `roomId`
   - тот же `eventType = m.room.message`
   - тот же body
   - тот же `txnId`

3. Отправить запрос первый раз и сохранить:
   - HTTP-статус
   - response body
   - `event_id`, если он есть в ответе

4. Повторно отправить тот же запрос с тем же `txnId`.

5. Сравнить результаты:
   - повторный запрос не должен создать новое событие
   - второй ответ должен логически соответствовать первому
   - в идеале должен вернуться тот же `event_id`

6. Дополнительно проверить состояние ленты:
   - в интерфейсе должно отображаться только одно сообщение
   - дубликат сообщения не должен появиться

### Что ещё полезно проверить дополнительно

Отдельно я бы проверил обратный случай:
- `roomId` тот же
- body тот же
- `txnId` уже другой

В этом случае система должна обработать запрос как новую отправку.

Такая проверка помогает убедиться, что идемпотентность завязана именно на `txnId`, а не просто на одинаковый текст сообщения.

---

## Итог

Для данного сценария я бы считал API-часть корректной, если выполняются три условия:
1. валидный авторизованный запрос успешно создаёт событие сообщения
2. невалидные условия, например неверная комната или невалидный токен, приводят к понятной ошибке и не создают сообщение
3. повторный запрос с тем же `txnId` не создаёт дубликат и возвращает тот же логический результат
